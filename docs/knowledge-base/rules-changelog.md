| 2026-04-18 | MKT_001 Phase 4 Terraform | Never run `aws s3api create-bucket` without verifying the bucket appeared in your account via `aws s3 ls` immediately after — bucket names are globally unique; a silently-failed create leaves `terraform init` routing to a stranger's bucket in another region, producing confusing 301/403 AccessDenied errors. |
| 2026-04-18 | MKT_001 Phase 4 | Never use a `from` address whose domain isn't exactly the verified Resend sending domain — subdomains are verified separately from their apex; verifying `send.example.com` does NOT authorize sends from `@example.com`. Mismatch produces Lambda returns 200 (DDB write succeeded), notification silently dropped at Resend, visible only in CloudWatch logs as `Resend error 403 domain not verified`. |
| 2026-04-18 | MKT_001 Phase 5 | Never plan S3-behind-Cloudflare via REST endpoint (`s3.*.amazonaws.com`) on Cloudflare Free plan. Cloudflare proxied origin connections send the proxied hostname as SNI (e.g. `housemoneyportfolio.com`), not the S3 bucket hostname. S3 rejects this under TLS — Cloudflare "Full (strict)" fails. Fix requires Host Header Override / Origin Rules (Cloudflare Pro, $20/mo). For Free-plan: use S3 website endpoint (`s3-website.*.amazonaws.com`) + SSL mode "Full" + public bucket policy. Revisit via Cloudflare Worker (OPS_003). |
| 2026-04-18 | MKT_001 Phase 5 | Never name an S3 bucket something other than the exact custom domain when using S3 static website hosting with a custom domain via Cloudflare/CDN proxy. S3 website endpoint uses the incoming Host header to look up the bucket — bucket name must exactly equal the Host header value. For housemoneyportfolio.com, the bucket must be named housemoneyportfolio.com. Non-matching bucket names (e.g., hmp-website-prod) produce NoSuchBucket errors. Cloudflare Pro's Origin Rules can override the Host header, but on Free plan the only fix is to match bucket name to domain. |
| 2026-04-18 | MKT_001 Phase 5 | Never add `trailingSlash: true` to `next.config.mjs` while App Router image metadata files (`app/icon.svg`, `app/apple-icon.png`, etc.) are present — in this project's Next.js 14 setup, trailing-slash mode produces `PageNotFoundError` on those generated routes during build. Fix: move metadata files to `public/` with standard names (`icon.svg`, `apple-touch-icon.png`) and add explicit `<link>` tags in `layout.tsx`. This also produces the correct `privacy/index.html` output structure for S3 website hosting without a Cloudflare Transform Rule. |
| 2026-04-19 | FIX_001 | Never use Lambda Function URLs (`AuthType=NONE`) for any public endpoint on AWS account `897545367327`. The account has a server-side block that returns `403 AccessDeniedException` on every invocation regardless of resource policy. Affects every Function URL on every Lambda function in the account. Failure mode: 403 with "For troubleshooting Function URL authorization issues" message; zero CloudWatch log entries because the handler is never invoked; not visible to `get-policy`/`get-function-url-config`/`get-account-settings`/`organizations describe-organization` (standalone account, no SCPs); not exposed via CLI v2.34.32. Default to API Gateway HTTP API for public Lambda invocation. Discovered after MKT_001 Phase 5 went live — the form had been broken since Phase 4 deploy because Phase 4 verification used `aws lambda invoke` only. |
| 2026-04-19 | FIX_001 | Never verify a Lambda Function URL or any public HTTPS endpoint by `aws lambda invoke` alone. Direct management-API invocation bypasses the endpoint's resource-based policy and auth layer entirely. Always test with a real HTTP request (curl to `*.lambda-url.*.on.aws` / `*.execute-api.*.amazonaws.com` / your custom domain, or a browser form submission) before declaring the endpoint working end-to-end. MKT_001 Phase 4 claimed "end-to-end verified" based on `aws lambda invoke` output; the URL's auth layer was never exercised until Phase 5 went live and a real form submission hit the 403. The Function URL had been dead since the moment it was created. |
| 2026-04-19 | FIX_001 | Never assume AWS CLI feature availability equals AWS API/service feature availability. Some AWS features (Lambda's Public Access Block, Resource Control Policies, newer setting APIs) may exist at the service layer but not be exposed in older CLI versions. If a setting is blocking behavior but no CLI command exists in the installed version, consult AWS documentation directly — the AWS Console or a newer CLI may expose the setting. When the block can't be cleared from your side, treat it as an account-level constraint and pivot the architecture rather than burning hours probing for the toggle. |
| 2026-04-20 | FIX_001 Phase C | Never assume `DomainStatus: AVAILABLE` on a new API Gateway regional custom domain means it's fully propagated. AWS documents up to 40 minutes for full propagation across regional edge nodes. AWS regional endpoints (`d-<hash>.execute-api.<region>.amazonaws.com`) are geo-balanced via DNS — different resolvers receive different IP pools. A curl from one location may succeed while a curl from another (Cloudflare's edge, for example) still hits an unprovisioned node. When 521s/connection-refused appear post-creation, wait the full documented window before declaring a different failure mode. |
| 2026-04-20 | FIX_001 Phase C | Never proxy AWS API Gateway regional custom domains through Cloudflare on the Free plan without first verifying the proxied path works end-to-end. Cloudflare Free → APIGW regional endpoint can return persistent 521 (connection refused) even when the origin is fully healthy and reachable from any non-Cloudflare client. Root cause undetermined — likely a Cloudflare-side edge IP behavior or Free-plan SSL/SNI handling specific to APIGW regional endpoints. Workaround: set the API CNAME to DNS-only (grey cloud); accept the loss of WAF/Bot Fight Mode and rate limiting. Add Cloudflare Turnstile to the form for client-side bot mitigation as a substitute (free, ~equivalent abuse protection). Tracked: OPS_005 (proxy restoration investigation) and OPS_006 (Turnstile implementation). |
| 2026-04-20 | MKT_003 | Never run `npm run build` while the operator's `npm run dev` server is active. `next build` writes to the same `.next` directory the dev server reads from, which breaks the dev server's hot-reload watcher state and forces the operator to restart it. Symptom: operator says "it's not working locally anymore" mid-session shortly after a build. Mid-session verification should use `npx tsc --noEmit` (read-only, doesn't touch `.next`). If `npm run build` is genuinely required for an AC, warn the operator first so they can stop their dev server, or schedule the build for after the visual check is complete. |
